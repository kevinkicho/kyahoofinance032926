Design — Section 3: Data Architecture & Build Sequence

How mock data is structured, and the recommended order to build each sub-project

Data Layer Pattern (per market)

Each market module owns its data file. Mock data is shaped to exactly match the real API it will eventually consume — so the swap is a one-line import change.

src/markets/bonds/
  BondsMarket.jsx          ← root component + state
  components/
    YieldCurve.jsx
    CreditMatrix.jsx
    SpreadMonitor.jsx
    DurationLadder.jsx
  data/
    mockBondsData.js       ← shaped like FRED API response
    useBondsData.js        ← hook: returns mock now, real API later
The swap rule: when ready for real data, only useBondsData.js changes — it fetches from FRED instead of importing the mock file. All components are untouched.

Real API targets (for future swap-in)

🏛️ Bonds
FRED API (free) — yield series, sovereign spreads
api.stlouisfed.org/fred
💱 FX
Frankfurter (already live ✓) + ECB for history
frankfurter.app — already in useFrankfurterRates
📊 Derivatives
Yahoo Finance options chain (existing /api/stocks)
Extend server/index.js with /api/options/:ticker
🏠 Real Estate
FRED (Case-Shiller index, housing starts)
Same FRED API as bonds — reuse server proxy
🛡️ Insurance
No free real-time API — curated static dataset. Company data from Yahoo Finance (existing). Industry aggregates from NAIC public data (annual).
