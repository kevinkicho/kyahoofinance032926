Hub Redesign — 3 Implementation Approaches

How should we restructure the existing app into a multi-market hub?

A

Market Router — top-level routing per market

Introduce a lightweight client-side router (or manual state) where the active market is the top-level state. Each market is a self-contained module: its own tab config, sub-views, data files, and sidebar. App.jsx becomes a thin shell that renders <MarketRouter> → <EquitiesMarket> / <BondsMarket> / etc. The existing equities code moves into src/markets/equities/ with minimal changes.

src/markets/
  equities/ ← existing code moves here
  bonds/
  fx/
  derivatives/
  realEstate/
  insurance/
src/hub/ ← tab bar, market switcher
App.jsx ← thin shell (~30 lines)

Pros

  • Clean isolation per market
  • Easy to add markets later
  • Equities code barely changes

Cons

  • More directories to navigate
B

Expand App.jsx — add markets as new view modes

Keep the existing structure. Add bonds, fx, etc. as additional values of viewMode. The tab bar replaces the Header's current view buttons. Each new market is a new top-level component rendered in the existing <main> area. Minimal structural change — just more cases in the existing switch.

App.jsx ← grows from 547 → ~900+ lines
components/
  BondsView/
  FXView/
  ... (flat, alongside equities)

Pros

  • Lowest disruption to existing code
  • Familiar pattern

Cons

  • App.jsx becomes very large
  • All market state in one component
  • Harder to reason about later
C

Context + Registry — plugin-style market system

Define a MarketRegistry — an array of market config objects, each declaring its id, label, icon, sub-views, and root component. The hub reads the registry to render tabs and route views. Adding a new market is purely additive: register a config object, drop in a component. Uses React Context for shared hub state (active market, currency, snapshot date).

src/hub/MarketRegistry.js ← array of market configs
src/hub/HubContext.js ← shared state
src/markets/equities/index.jsx
src/markets/bonds/index.jsx
... (each self-contained)

Pros

  • Most extensible long-term
  • Clean plugin mental model
  • Hub logic fully decoupled from markets

Cons

  • Most upfront architecture work
  • Slight over-engineering for 6 known markets
Recommendation

A (Market Router) — best balance of clean architecture and build effort. Each market is isolated (easy to reason about, test, and extend), equities code barely changes, and App.jsx stays thin. C is slightly more elegant but adds complexity not yet justified by 6 fixed markets.
